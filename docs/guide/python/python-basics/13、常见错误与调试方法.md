# 常见错误与调试方法

在 Python 编程过程中，开发者常会遇到各类错误。掌握常见错误类型及其调试方法，是提升开发效率和代码质量的关键。本章介绍两类核心调试手段：`print` 调试（快速验证）与 `logging` 模块（结构化日志记录）。

------

## 一、常见错误类型

| 错误类型            | 触发场景示例                            | 典型信息特征                                               |
| ------------------- | --------------------------------------- | ---------------------------------------------------------- |
| `SyntaxError`       | 语法错误，如缺少冒号、括号不匹配        | `SyntaxError: invalid syntax`                              |
| `NameError`         | 使用未定义的变量                        | `NameError: name 'x' is not defined`                       |
| `TypeError`         | 类型不匹配，如对整数调用 `.lower()`     | `TypeError: 'int' object is not callable`                  |
| `ValueError`        | 值合法但不符合函数要求，如 `int("abc")` | `ValueError: invalid literal for int()`                    |
| `IndexError`        | 列表/元组索引越界                       | `IndexError: list index out of range`                      |
| `KeyError`          | 字典中访问不存在的键                    | `KeyError: 'missing_key'`                                  |
| `AttributeError`    | 对象没有指定属性或方法                  | `AttributeError: 'list' object has no attribute 'appendx'` |
| `FileNotFoundError` | 尝试打开不存在的文件                    | `FileNotFoundError: [Errno 2] No such file`                |

> 注意：所有异常均继承自 `BaseException`，绝大多数用户代码应捕获 `Exception` 及其子类。

------

## 二、调试方法对比

| 方法      | 适用阶段             | 优点                             | 缺点                         | 推荐场景                       |
| --------- | -------------------- | -------------------------------- | ---------------------------- | ------------------------------ |
| `print`   | 快速原型、简单脚本   | 简单直观，无需额外配置           | 难以管理，上线需删除，无分级 | 临时排查、教学演示             |
| `logging` | 中大型项目、生产环境 | 支持日志级别、格式、输出目标分离 | 需要初始化配置               | 长期维护、多模块协作、部署系统 |

------

## 三、`print` 调试示例

适用于快速验证变量值或执行路径。

```python
def divide(a, b):
    print(f"[DEBUG] a={a}, b={b}")  # 调试输出
    if b == 0:
        print("[DEBUG] 除数为零，即将抛出异常")
        raise ValueError("除数不能为零")
    result = a / b
    print(f"[DEBUG] 计算结果: {result}")
    return result

# 测试
try:
    divide(10, 0)
except ValueError as e:
    print(f"捕获异常: {e}")
```

**注意事项**：

- 调试完成后应删除或注释掉 `print` 语句。
- 避免在循环中频繁打印，以免影响性能或淹没关键信息。
- 不应用于生产环境日志记录。

------

## 四、`logging` 模块简介

`logging` 是 Python 标准库提供的结构化日志工具，支持多级别、多输出、格式定制。

### 4.1 日志级别（由低到高）

| 级别       | 函数调用            | 用途说明                       |
| ---------- | ------------------- | ------------------------------ |
| `DEBUG`    | `logger.debug()`    | 详细信息，通常仅用于诊断       |
| `INFO`     | `logger.info()`     | 确认程序按预期工作             |
| `WARNING`  | `logger.warning()`  | 表示潜在问题，但程序仍正常运行 |
| `ERROR`    | `logger.error()`    | 发生严重问题，部分功能失效     |
| `CRITICAL` | `logger.critical()` | 严重错误，程序可能无法继续运行 |

默认级别为 `WARNING`，低于该级别的日志不会输出。

### 4.2 基础配置与使用

```python
import logging

# 配置日志格式和级别（通常在程序入口处调用一次）
logging.basicConfig(
    level=logging.DEBUG,
    format='%(asctime)s - %(levelname)s - %(message)s',
    datefmt='%Y-%m-%d %H:%M:%S'
)

def safe_divide(a, b):
    logging.debug(f"调用 safe_divide({a}, {b})")
    try:
        if b == 0:
            logging.error("除数为零，操作无效")
            raise ValueError("除数不能为零")
        result = a / b
        logging.info(f"计算成功: {a} / {b} = {result}")
        return result
    except Exception as e:
        logging.exception("发生未预期的异常")  # 自动记录 traceback
        raise

# 测试
if __name__ == "__main__":
    try:
        safe_divide(10, 0)
    except ValueError:
        logging.warning("主程序捕获了除零错误")
```

**输出示例**：

```
2026-03-01 22:58:00 - DEBUG - 调用 safe_divide(10, 0)
2026-03-01 22:58:00 - ERROR - 除数为零，操作无效
2026-03-01 22:58:00 - ERROR - 发生未预期的异常
Traceback (most recent call last):
  File "...", line 12, in safe_divide
    raise ValueError("除数不能为零")
ValueError: 除数不能为零
2026-03-01 22:58:00 - WARNING - 主程序捕获了除零错误
```

### 4.3 关键注意事项

- **配置一次即可**：`basicConfig()` 仅在 root logger 未配置时生效，重复调用无效。
- **避免在库中调用 `basicConfig()`**：库应使用 `logger = logging.getLogger(__name__)`，由主程序统一配置。
- **使用 `logger.exception()`**：在 `except` 块中记录异常时，自动包含堆栈跟踪。
- **日志不影响程序逻辑**：即使日志级别过滤掉某条消息，其字符串拼接仍会执行，高频率场景建议使用延迟求值（如 `logger.debug("Value: %s", expensive_func())`）。

------

## 五、调试最佳实践总结

1. **优先使用 `logging`**：对于超过 100 行的脚本或需长期维护的项目，应使用 `logging`。
2. **合理设置日志级别**：开发时设为 `DEBUG`，生产环境通常设为 `INFO` 或 `WARNING`。
3. **不要混用 `print` 和 `logging`**：保持日志输出方式统一。
4. **记录上下文信息**：日志消息应包含足够上下文（如函数名、关键变量），便于定位问题。
5. **异常必须记录**：所有 `except` 块应至少记录 `ERROR` 级别日志，防止“静默失败”。

通过结合对错误类型的识别与合适的调试工具，可显著提升问题排查效率和代码健壮性。
